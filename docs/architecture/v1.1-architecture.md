# Relics Platform V1.1 Architecture

Date: 2026-05-16
Version: 1.1.0

## Architecture Goals

V1.1 is an architecture refactor release. It keeps the product behavior compatible with V1.0.0 while reducing oversized modules, clarifying ownership boundaries, adding tests and CI, and simplifying daily startup.

The daily development workflow now has two primary entry points:

- `start-backend.bat` starts the FastAPI backend.
- `start-frontend.bat` starts both frontend development servers.

Data pipeline scripts remain available through `platform/scripts/run_pipeline.py`, legacy numbered BAT files, and the Admin pipeline UI. They no longer need to be started as part of every ordinary frontend/backend development session.

## Backend Boundaries

`platform/webgis/main.py` is now the composition root. It wires configuration, FastAPI lifespan, static mounts, legacy templates and API routers.

`tile_routes.py` owns map tile proxying, tile cache inspection and offline tile download APIs.

`terrain_routes.py` owns local DEM terrain tile routes and delegates raster tile generation to `terrain_provider.py`.

`routers/admin.py` now focuses on pipeline status, pipeline detail items, task execution, DOCX upload and single-file processing.

`routers/admin_task_service.py` owns task registry, script aliases, subprocess lifecycle, log collection and duplicate-run protection.

`routers/admin_relic_routes.py` owns Admin relic CRUD, audit, import/export, dashboard stats, dictionary endpoints and batch operations.

`data_loader.py` remains the compatibility facade for `store.*`, while query-heavy and helper-heavy code moved into:

- `data_serializers.py` for SQLite row to legacy payload mapping.
- `survey_coverage.py` for survey GPS parsing and village coverage.
- `data_admin_queries.py` for Admin list, nearby query, export iteration and township reads.
- `data_admin_stats.py` for Admin dashboard aggregates and legacy stats.

## Pipeline Boundaries

`platform/scripts/run_pipeline.py` is the pipeline orchestrator. It supports:

- `--list` to print registered steps.
- `--dry-run` to preview execution.
- `--skip` to skip one or more steps.

The default pipeline contains seven steps:

1. `step01_convert_docs.py`: DOCX to structured Markdown.
2. `step02_build_dataset.py`: Markdown to CSV, JSON and GeoJSON.
3. `step03_extract_photos.py`: extract embedded archive photos.
4. `step04_extract_drawings.py`: extract embedded drawings.
5. `step05_convert_worklogs.py`: worklog spreadsheets to PDF.
6. `step06_prepare_boundaries.py`: boundary files to WGS-84 GeoJSON.
7. `step07_build_db.py`: build SQLite `relics.db` with R-Tree, FTS5 and audit tables.

## Frontend Boundaries

React WebGIS remains the primary map experience under `/app/`. It owns Cesium rendering, three.js model viewing, user-facing panels, filtering and map interaction.

Vue Admin remains the operational console under `/admin-ui/`. In V1.1:

- `Dashboard.vue` is reduced to the view shell.
- `useDashboard.ts` owns dashboard state and actions.
- `dashboard.css` owns dashboard presentation.
- `RelicEditDialog.vue` and `Relics.vue` have externalized styles, leaving behavior intact.

## Testing And CI

The backend has pytest coverage for pipeline orchestration, Admin task service, data helper delegation and Admin router composition.

GitHub Actions runs Python tests and frontend validation. Local verification should use:

```powershell
.venv\Scripts\python.exe -m pytest
npm.cmd run type-check --prefix platform\webgis-react
npm.cmd run build --prefix platform\webgis-react
npm.cmd run typecheck --prefix platform\admin-vue
npm.cmd run build --prefix platform\admin-vue
```

## Release Notes

V1.1.0 is an architecture release. It is intended to be source-compatible with V1.0.0 data conventions while improving maintainability, startup ergonomics and automated verification.
