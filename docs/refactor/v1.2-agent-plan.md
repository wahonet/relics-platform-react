# V1.2 Agent 执行计划

日期：2026-05-21
状态：P0 首轮已落地，P1/P2 待执行

本文面向未来接手本仓库的 agent。目标不是重写平台，而是在 V1.1 架构调整的基础上，把后台、后端服务层、数据管线和生产可用性继续推进到更稳定的状态。

## 读取顺序

未来 agent 开工前先读这些文件：

1. `README.md`：确认启动方式、数据目录和当前功能边界。
2. `docs/architecture/v1.1-architecture.md`：确认模块边界。
3. `docs/refactor/execution-log.md`：确认 V1.1 已经做过什么，避免重复改造。
4. 本文件：按任务包执行。

## 执行原则

- 保持 V1.0/V1.1 现有 API 和数据目录兼容，除非有明确迁移说明。
- 每次只做一个任务包，避免把后端、React、Vue 和管线同时大改。
- 优先补测试，再做高风险重构。
- 不删除 legacy 静态页面，除非先确认 React/Vue 已覆盖同等功能并更新文档。
- 所有新增配置写入 `config.example.yaml`，并保证缺省值可本地启动。
- 文档同步更新：改启动、端口、目录、配置、命令时，至少更新 `README.md` 和对应 docs。

后端改动至少运行：

```powershell
.venv\Scripts\python.exe -m pytest
.venv\Scripts\python.exe platform\scripts\run_pipeline.py --list
.venv\Scripts\python.exe platform\scripts\run_pipeline.py --dry-run --skip 01 --skip 05
```

React WebGIS 改动至少运行：

```powershell
npm.cmd run type-check --prefix platform\webgis-react
npm.cmd run build --prefix platform\webgis-react
```

Vue Admin 改动至少运行：

```powershell
npm.cmd run typecheck --prefix platform\admin-vue
npm.cmd run build --prefix platform\admin-vue
```

跨模块或 CI 相关改动运行全部命令，并检查：

```powershell
git diff --check
```

## P0：稳定边界与可维护性

执行状态：2026-05-21 已完成首轮实现。后续 agent 可继续细化，但不要重复搬迁同一批逻辑。

### 任务包 1：后端服务层收束

目标：减少 router 直接承载业务逻辑，让后端形成 `router -> service -> data/helper` 的结构。

首轮落地：

- 新增 `platform/webgis/services/admin_relic_service.py`。
- 已迁出字段标准化、bbox 解析、批量 payload 校验、CSV 导出、CSV/JSON 导入解析和导入执行。
- 新增 `tests/test_admin_relic_service.py`。

建议改动范围：

- `platform/webgis/routers/admin_relic_routes.py`
- `platform/webgis/routers/admin.py`
- `platform/webgis/routers/admin_task_service.py`
- 新增 `platform/webgis/services/`
- 视情况新增 `tests/test_*_service.py`

执行步骤：

1. 新建 `platform/webgis/services/`，先放文物、审计、导入导出中最容易剥离的纯业务函数。
2. 保持 router 的 URL、请求参数和响应结构不变。
3. 为剥离出来的 service 补单测。
4. 确认 `tests/test_admin_router_composition.py` 仍通过。

验收标准：

- router 文件只负责参数接收、鉴权/错误转换和 response。
- 文物 CRUD、批量操作、审计查询至少有一部分进入 service。
- 现有 Admin Vue 不需要跟着改接口。

### 任务包 3：管线产物校验

目标：让 7 步管线不仅能跑，还能判断产物是否可信。

首轮落地：

- `run_pipeline.py --dry-run` 已输出每步输入和产物状态。
- 实际运行会写入 `data/output/logs/pipeline_manifest.json`。
- 新增 `tests/test_pipeline_validation.py`。

建议改动范围：

- `platform/scripts/run_pipeline.py`
- `platform/scripts/_common.py`
- 各 `platform/scripts/step*.py`
- `tests/test_pipeline_orchestrator.py`
- 新增 `tests/test_pipeline_validation.py`

执行步骤：

1. 为每个步骤定义输入、输出、可选性和最小产物校验。
2. 在 `--dry-run` 输出中显示每步输入是否存在、输出是否存在、是否建议跳过。
3. 管线完成后生成 `data/output/logs/pipeline_manifest.json`，记录步骤、开始/结束时间、产物路径、数量和错误。
4. Admin 管线工作台后续可读取 manifest，但本任务包先不强制改 Vue。

验收标准：

- `run_pipeline.py --dry-run` 不依赖真实业务数据也能输出稳定结果。
- 单测覆盖 manifest 生成和缺失输入提示。
- 旧的 `--list`、`--skip` 行为不破坏。

## P1：生产化能力

### 任务包 4：认证与权限增强

目标：从演示登录升级为可局域网部署的基础权限体系。

建议改动范围：

- `config.example.yaml`
- `platform/webgis/main.py`
- `platform/webgis/routers/admin.py`
- `platform/webgis/routers/admin_relic_routes.py`
- `platform/admin-vue/src/stores/auth.ts`
- `platform/admin-vue/src/views/Login.vue`
- 新增 `tests/test_auth_permissions.py`

执行步骤：

1. 保留 `server.enable_auth: false` 的演示语义。
2. `enable_auth: true` 时支持用户角色：`admin`、`editor`、`viewer`。
3. 写操作要求 `admin` 或 `editor`，审计/导出等敏感能力按角色限制。
4. 密码不要继续明文生产化，新增兼容字段时保证旧模板还能启动。

验收标准：

- 关闭鉴权时本地开发体验不变。
- 开启鉴权时不同角色访问受限接口有测试覆盖。
- README 明确生产部署必须修改默认账号和密码。

### 任务包 5：数据库迁移机制

目标：降低真实业务数据升级时重建 SQLite 的风险。

建议改动范围：

- `platform/scripts/step07_build_db.py`
- 新增 `platform/webgis/db/` 或 `platform/scripts/db_migrations/`
- 新增 `tests/test_db_migrations.py`

执行步骤：

1. 在 SQLite 中维护 `schema_version` 表。
2. 将建表 SQL 和索引 SQL 拆成可版本化 migration。
3. 支持从旧版本升级到当前版本。
4. 保留从 CSV/JSON 全量重建数据库的能力。

验收标准：

- 空库可初始化。
- 旧 schema fixture 可升级。
- R-Tree、FTS5、audit_log 等关键表升级后可用。

### 任务包 6：数据质量报告

目标：让导入前后能发现档案数据问题。

建议改动范围：

- `platform/scripts/step02_build_dataset.py`
- `platform/webgis/data_admin_stats.py`
- `platform/webgis/routers/admin_relic_routes.py`
- `platform/admin-vue/src/views/Import.vue`
- `platform/admin-vue/src/views/Dashboard.vue`

- 后台能看到问题摘要。

## 暂缓事项

这些事情不要作为第一批任务：

- 全面替换 FastAPI、React 或 Vue 技术栈。
- 删除 `platform/webgis/templates/` legacy 页面。
- 引入大型任务队列、微服务或复杂部署平台。
- 对真实业务字段做不可逆重命名。
- 大规模视觉重设计。

## 推荐提交节奏

每个任务包建议一个 PR 或一个清晰 commit：

1. 先提交测试或 fixture。
2. 再提交实现。
3. 最后提交文档更新。

提交说明建议包含：

- 改了哪些模块。
- 保持了哪些兼容性。
- 跑过哪些验证命令。
- 是否需要用户更新 `config.yaml` 或重新跑管线。
